3. 번역 (Translation)

이제 번역(translation)에 대해서 알아봅시다. 이것은 또 다른 형태의 sequence-to-sequence 태스크입니다. 즉, 한 시퀀스에서 다른 시퀀스로 이동하는(변형하는) 것으로 공식화할 수 있는 문제입니다. 그런 의미에서 이 문제는 요약(summarization)과 매우 유사하고 여기에서 보게 될 내용을 다음과 같은 다른 sequence-to-sequence 문제에 적용할 수 있습니다:

  • 스타일 트랜스퍼(style transfer): 특정 스타일로 작성된 텍스트를 다른 스타일로 번역하는 모델 생성(예: 격식 스타일에서 캐주얼 스타일로 또는 셰익스피어 영어에서 현대 영어로)

  • 생성 기반 질의 응답(generative question answering): 주어진 맥락(context)에서 질문에 대한 답변을 생성하는 모델 만들기

두 개(또는 그 이상) 언어로 된 충분히 큰 텍스트 코퍼스가 있는 경우 인과적 언어 모델링(causal language modeling) 섹션에서 하는 것처럼 처음부터 새로운 번역 모델을 학습할 수 있습니다. 그러나 일정 규모의 특정 언어로 표현된 말뭉치를 가지고 mT5나 mBART 등과 같은 이미 존재하는 다국어 번역 모델을 미세 조정하는 편이 훨씬 빠를 겁니다.

이 섹션에서는 KDE 앱용으로 현지화된 파일 데이터셋인 KDE4 데이터셋을 이용하여 영어에서 프랑스어로 번역하도록 사전 학습된 Marian 모델을 미세 조정할 것입니다. 우리가 사용할 모델은 실제로 KDE4 데이터셋이 포함된 Opus 데이터셋에 서 가져온 대규모 프랑스어 및 영어 텍스트 코퍼스에서 사전 학습되었습니다. 그러나 우리가 사용하는 사전 학습된 모델이 학습 과정에서 해당 데이터셋을 이미 학습했다고 하더라도 미세 조정 후에 더 나은 버전을 얻을 수 있음을 알 수 있습니다.

완료되면 다음과 같은 예측을 할 수 있는 모델을 도출할 수 있습니다:

이전 섹션에서와 같이 아래 코드를 사용하여 학습하고 Hub에 업로드할 실제 모델을 찾고 여기에서 예측 결과를 다시 확인할 수 있습니다.

데이터 준비

번역 모델을 미세 조정하거나 처음부터 사전 학습하려면 작업에 적합한 데이터셋이 필요합니다. 이전에 언급했듯이 이 섹션에서는 KDE4 데이터셋을 사용하지만 번역하려는 두 언어의 문장 쌍이 있는 한 자체 데이터를 사용하도록 코드를 매우 쉽게 조정할 수 있습니다. Dataset을 이용하여 사용자 정의 데이터를 로드하는 방법에 대한 복습이 필요한 경우 5장을 다시 살펴보세요.

KDE4 데이터셋

이전과 같이 load_dataset() 함수를 사용하여 데이터셋을 다운로드합니다:

from datasets import load_dataset, load_metric

raw_datasets = load_dataset("kde4", lang1="en", lang2="fr")

다른 언어 쌍으로 작업하려는 경우 원하는 언어 코드를 지정하여 해당 데이터셋을 다운로드할 수 있습니다. 이 데이터셋에서는 총 92개 언어를 사용할 수 있습니다. 데이터셋 카드의 언어 태그를 확장하면 모두 볼 수 있습니다.

다운로드한 데이터셋을 한번 살펴보겠습니다.

raw_datasets

총 210,173건의 문장 쌍을 포함하고 있지만 단일 분할이므로 여기서 검증 집합을 별도로 생성해야 합니다. 5장에서 보았듯이 Dataset에는 이 작업을 쉽게 수행할 수 있는 train_test_split() 메서드가 있습니다:

split_datasets = raw_datasets["train"].train_test_split(train_size=0.9, seed=20)
split_datasets

다음과 같이 "test" 키의 이름을 "validation"으로 바꿀 수 있습니다:

split_datasets["validation"] = split_datasets.pop("test")

이제 데이터셋의 한 요소를 살펴보겠습니다:

split_datasets["train"][1]["translation"]

위와 같이 우리가 요구한 언어쌍(영어, 불어)에 해당하는 문장들이 포함된 딕셔너리가 반환됩니다. 기술적인 컴퓨터 과학 용어로 가득 찬 이 데이터셋의 한 가지 특징은 모두 프랑스어로 완전히 번역되었다는 것입니다. 그러나 프랑스 엔지니어들은 종종 게을러서 말할 때 대부분의 컴퓨터 과학 관련 단어를 영어로 표현하고 있습니다. 예를 들어, 위 예시에서 "threads"라는 단어는 불어 문장, 특히 기술적인 대화에서 많이 나타납니다. 그러나 이 데이터 세트에서는 더 "회의를 위한 토론(fils de discussion)"으로 정확하게 번역되었습니다. 규모가 더 큰 프랑스어 및 영어 문장 말뭉치에서 사전 학습된 아래와 같은 모델은 단어를 번역하지 않고 있는 그대로 그냥 두는 더 쉬운 옵션을 사용합니다:

from transformers import pipeline

model_checkpoint = "Helsinki-NLP/opus-mt-en-fr"
translator = pipeline("translation", model=model_checkpoint)
translator("Default to expanded threads")

또 다른 예는 "plugin"이라는 단어에서 볼 수 있습니다. 이 단어는 공식적으로 프랑스어가 아니지만 대부분의 원어민이 이해하고 번역하는데 어렵지 않을 것입니다. KDE4 데이터셋에서 이 단어는 프랑스어로 더 공식적인 "module d'extension"으로 번역되었습니다:

split_datasets["train"][172]["translation"]

그러나 위의 사전 학습된 모델은 간결하고 친숙한 영어 단어를 고수합니다:

translator(
    "Unable to import %1 using the OFX importer plugin. This file is not the correct format."
)

미세 조정된 모델이 데이터셋의 이러한 특성을 흡수하는지 확인하는 것은 흥미로울 것입니다.

✏️ Your turn! 프랑스어에서 자주 사용되는 또 다른 영어 단어는 "email"입니다. 이 단어를 사용하는 학습 데이터셋에서 첫 번째 샘플을 찾아봅시다. 어떻게 번역됩니까? 사전 학습된 모델은 동일한 영어 문장을 어떻게 번역합니까?

데이터 처리

지금쯤이면 기본적인 절차를 알아야 합니다. 모든 텍스트는 모델이 이해할 수 있도록 토큰 ID 세트로 변환되어야 합니다. 이 작업을 위해 입력과 타겟을 모두 토큰화해야 합니다. 첫 번째 작업은 tokenizer 객체를 만드는 것입니다. 앞서 언급했듯이 Marian English to French 사전 학습 모델을 사용할 것입니다. 다른 언어 쌍에 대해서 작업하려면 모델 체크포인트를 변경해야 합니다. Helsinki-NLP 조직은 여러 언어로 천 개 이상의 모델을 제공합니다.

from transformers import AutoTokenizer

model_checkpoint = "Helsinki-NLP/opus-mt-en-fr"
tokenizer = AutoTokenizer.from_pretrained(model_checkpoint, return_tensors="pt")

Hub에서 선호하는 다른 모델로 model_checkpoint를 바꾸거나 사전 학습된 모델과 토크나이저를 저장한 로컬 폴더로 바꿀 수도 있습니다.

💡 mBART, mBART-50 또는 M2M100과 같은 다국어 토크나이저를 사용하는 경우 tokenizer.src_lang 및 tokenizer.tgt_lang을 올바른 값으로 설정하여 토크나이저에서 입력 및 대상의 언어 코드를 설정해야 합니다.

데이터 준비 과정은 매우 직관적입니다. 기억해야 할 것이 하나 있습니다. 평소와 같이 입력을 처리하지만 타겟(target)의 경우 컨텍스트 관리자 내부의 토크나이저를 as_target_tokenizer()로 래핑해야 합니다.

Python의 컨텍스트 관리자는 with 문과 함께 도입되었으며 두 개의 관련 작업을 쌍으로 실행할 때 유용합니다. 이에 대한 가장 일반적인 예는 파일을 쓰거나 읽을 때이며, 이는 종종 다음과 같은 명령어 내에서 수행됩니다:

# with open(file_path) as f:
#     content = f.read()

여기서 쌍으로 실행되는 두 가지 관련 작업은 파일을 열고 닫는 작업입니다. 열린 파일 f에 해당하는 객체는 with 아래의 들여쓰기된 블록 내부에만 존재합니다. 열기는 해당 블록보다 먼저 발생하고 블록 끝에서 닫기가 발생합니다.

따라서 하나의 샘플을 전처리하는 과정은 다음과 같습니다:

en_sentence = split_datasets["train"][1]["translation"]["en"]
fr_sentence = split_datasets["train"][1]["translation"]["fr"]

inputs = tokenizer(en_sentence)
with tokenizer.as_target_tokenizer():
    targets = tokenizer(fr_sentence)

컨텍스트 관리자 내에서 대상을 토큰화하는 것을 잊어버리면 입력 토크나이저에 의해 타겟(target)이 토큰화됩니다. 이럴 경우 Marian 모델의 경우에는 토크나이징이 제대로 동작하지 않을 겁니다:

wrong_targets = tokenizer(fr_sentence)
print(tokenizer.convert_ids_to_tokens(wrong_targets["input_ids"]))
print(tokenizer.convert_ids_to_tokens(targets["input_ids"]))

우리가 볼 수 있듯이, 영어 토크나이저를 사용하여 프랑스어 문장을 전처리하면 토크나이저는 프랑스어 단어를 모르기 때문에 훨씬 더 많은 토큰이 생성됩니다.

inputs과 targets은 모두 일반적인 키(input IDs, attention mask 등)가 있는 딕셔너리이므로 마지막 단계는 입력 내부에 "labels" 키를 설정하는 것입니다. 이 작업은 데이터셋에 적용할 전처리 함수에서 수행합니다:

max_input_length = 128
max_target_length = 128


def preprocess_function(examples):
    inputs = [ex["en"] for ex in examples["translation"]]
    targets = [ex["fr"] for ex in examples["translation"]]
    model_inputs = tokenizer(inputs, max_length=max_input_length, truncation=True)

    # 타겟을 위한 토크나이저 셋업
    with tokenizer.as_target_tokenizer():
        labels = tokenizer(targets, max_length=max_target_length, truncation=True)

    model_inputs["labels"] = labels["input_ids"]
    return model_inputs

입력 및 출력에 대해 동일한 최대 길이를 설정했습니다. 우리가 다루고 있는 텍스트는 매우 짧기 때문에 128을 사용합니다.

💡 T5 모델(더 구체적으로 t5-xxx 체크포인트 중 하나)을 사용하는 경우, 모델은 텍스트 입력에 translate: English to French:와 같이 당면한 작업을 나타내는 접두사가 있을 것으로 예상합니다.

⚠️ 타겟의 어텐션 마스크는 모델이 예상하지 못하기 때문에 주의를 기울이지 않습니다. 대신 패딩 토큰에 해당하는 레이블을 -100으로 설정해야 손실 계산에서 무시됩니다. 이 작업은 동적 패딩을 적용하기 때문에 나중에 데이터 콜레이터에 의해 수행되지만 여기에서 패딩을 사용하는 경우 패딩 토큰에 해당하는 모든 레이블을 -100으로 설정하도록 전처리 기능을 조정해야 합니다.

이제 데이터셋의 모든 분할에 대해 전처리를 한 번에 적용할 수 있습니다:

tokenized_datasets = split_datasets.map(
    preprocess_function,
    batched=True,
    remove_columns=split_datasets["train"].column_names,
)

이제 데이터가 전처리되었으므로 사전 학습된 모델을 미세 조정할 준비가 되었습니다!

Trainer API로 모델 미세 조정하기

Trainer를 사용하는 실제 코드는 약간만 변경하면 이전과 동일합니다. 여기에서는 Seq2SeqTrainer를 사용합니다. Seq2SeqTrainer는 Trainer의 하위 클래스로서 입력에 대한 출력을 예측하기 위해 generate() 메서드를 사용하여 평가를 적절하게 수행할 수 있습니다. 평가에 대한 내용은 메트릭(metric) 계산에 대해 이야기할 때 더 자세히 살펴보겠습니다.

먼저 미세 조정할 실제 모델이 필요합니다. 일반적인 AutoModel API를 사용합니다:

from transformers import AutoModelForSeq2SeqLM

model = AutoModelForSeq2SeqLM.from_pretrained(model_checkpoint)

모델 체크포인트가 번역 작업에 대해서 사전 학습된 모델을 나타내고 이를 그대로 사용하고 있으므로 누락된 가중치 또는 새로 초기화된 가중치에 대한 경고가 없습니다.

데이터 콜레이션 (Data Collation)

동적 배치 처리(Dynamic batching)를 위한 패딩을 처리하려면 데이터 콜레이터가 필요합니다. 이 경우 3장에서와 같이 DataCollatorWithPadding을 사용할 수 없습니다. 왜냐하면 이 메서드는 입력(input IDs, attention mask 및 token type IDs)에 대해서만 패딩을 수행하기 때문입니다. 레이블 역시 레이블에 있는 최대 길이로 채워져야 합니다. 그리고 앞서 언급했듯이 레이블을 채우는데 사용되는 패딩 값은 토크나이저의 패딩 토큰이 아니라 -100이어야 합니다. 그래야 패딩된 값이 손실 계산에서 무시됩니다.

이 작업은 모두 DataCollatorForSeq2Seq에 의해 수행됩니다. DataCollatorWithPadding과 마찬가지로, DataCollatorForSeq2Seq는 입력을 전처리하는데 사용되는 tokenizer는 물론 모델 자체도 매개변수로 입력받습니다. 모델도 입력받는 이유는 이 데이터 콜레이터가 시작 부분에 특수 토큰이 붙어 있는, 레이블 시퀀스를 우측으로 시프트(shift)한 버전인 디코더 input IDs를 준비하는 역할도 하기 때문입니다. 이 시프트(shift) 방법은 아키텍처마다 약간씩 다르기 때문에 DataCollatorForSeq2Seq는 모델 객체를 알아야 합니다.

from transformers import DataCollatorForSeq2Seq

data_collator = DataCollatorForSeq2Seq(tokenizer, model=model)

몇 개의 샘플에서 이를 테스트하기 위해 이 콜레이터에 토큰화된 학습 집합의 예제 목록을 입력해봅시다:

batch = data_collator([tokenized_datasets["train"][i] for i in range(1, 3)])
batch.keys()

-100을 사용하여 레이블이 배치의 최대 길이로 채워졌는지 확인할 수 있습니다:

batch["labels"]

또한 decoder_input_ids를 살펴보고 labels에 저장된 토큰 리스트의 시프트(shift) 버전인지 확인할 수 있습니다:

batch["decoder_input_ids"]

다음은 데이터셋의 첫 번째 및 두 번째 요소에 대한 레이블입니다:

for i in range(1, 3):
    print(tokenized_datasets["train"][i]["labels"])

이 data_collator를 Seq2SeqTrainer에 전달할 것입니다. 다음으로 평가 지표(metric)를 살펴보겠습니다.

평가지표 (Metrics)

Seq2SeqTrainer가 수퍼클래스 Trainer에 추가하는 기능은 평가(evaluation) 또는 예측(prediction) 중에 generate() 메서드를 사용하는 기능입니다. 학습하는 동안 모델은 학습 속도를 높이기 위해, 예측하려는 토큰 이후에 존재하는 토큰을 사용하지 않도록 하려고 어텐션 마스킹과 함깨 decoder_input_ids를 사용합니다. 추론하는 동안에는 레이블이 없기 때문에 이를 사용할 수 없으므로 동일한 설정으로 모델을 평가하는 것이 좋습니다.

1장에서 보았듯이 디코더는 토큰을 하나씩 예측하여 추론을 수행합니다. 이는 🤗Transformers에서 generate() 메서드에 의해 내부적으로 구현됩니다. predict_with_generate=True로 설정하면 Seq2SeqTrainer를 사용하여 위와 같은 방법으로 모델 평가를 수행할 수 있습니다.

번역에 사용되는 전통적인 평가지표(metric)는 BLEU 스코어로서 Kishore Papineni et al.에 의해 2002년 논문에 서 소개되었습니다. BLEU 점수는 번역 결과가 레이블에 얼마나 가까운지를 평가합니다. 모델에서 생성된 번역 결과의 명료성 또는 문법적 정확성을 측정하지는 않습니다. 단지 통계 규칙을 사용하여 생성된 출력 내의 모든 단어가 타깃(target)에도 존재하는지를 측정합니다. 또한, 타깃(target)에는 반복되지 않음에도 불구하고 번역 결과에서 동일한 단어가 계속 반복될 경우에도 패널티를 부여합니다. 이는 모델이 "the the the the the"와 같은 문장을 출력하는 것을 방지합니다. 또한 타깃(target)보다 짧은 문장을 출력하는 것에도 패널티를 부과합니다. 이는 "the"와 같은 문장이 출력되는 모델을 미연에 방지하기 위해서 입니다.

BLEU 점수의 한 가지 약점은 이미 토큰화된 텍스트를 대상으로 하기 때문에 다른 토크나이저를 사용하는 모델 간의 점수를 비교하기 어렵습니다. 따라서 오늘날 번역 모델을 벤치마킹하는데 가장 일반적으로 사용되는 평가지표(metric)는 토큰화 단계를 표준화하여 이러한 약점을 해결하는 SacreBLEU입니다. 이 평가지표를 사용하려면 먼저 SacreBLEU 라이브러리를 설치해야 합니다:

!python3 -m pip install sacrebleu

그런 다음 3장에서 했던 것처럼 load_metric()을 통해 로드할 수 있습니다.

from datasets import load_metric

metric = load_metric("sacrebleu")

이 메트릭은 텍스트를 입력(inputs) 및 타깃(targets)으로 사용합니다. 동일한 문장에 대해 허용 가능한 번역 결과가 여러 개 있기 때문에 여러 허용 가능한 대상을 입력받도록 설계되었습니다. 우리가 사용하는 데이터셋은 하나만 제공하지만 NLP에서 여러 문장을 레이블로 제공하는 데이터셋들도 많이 존재합니다. 따라서 예측(predictions)은 문장 리스트여야 하지만 정답(references)은 문장 리스트의 리스트라야 합니다.

예를 한번 봅시다:

predictions = [
    "This plugin lets you translate web pages between several languages automatically."
]

references = [
    [
        "This plugin allows you to automatically translate web pages between several languages."
    ]
]
metric.compute(predictions=predictions, references=references)

BLEU 점수가 46.75로 나왔고 그 점수가 나쁘지 않습니다. 참고로 "Attention Is All You Need" 논문의 원래 트랜스포머(Transformer) 모델은 영어와 프랑스어 간의 비슷한 번역 작업에서 BLEU 점수 41.8을 달성했습니다! counts 및 bp와 같은 개별 평가지표에 대한 자세한 내용은 SacreBLEU 저장소를 참조하세요. 반면에, 번역 모델에서 자주 나오는 두 가지 잘못된 유형의 예측(반복이 많거나 너무 짧음)으로 시도하면 다소 나쁜 BLEU 점수를 얻게 됩니다:

predictions = ["This This This This"]
references = [
    [
        "This plugin allows you to automatically translate web pages between several languages."
    ]
]
metric.compute(predictions=predictions, references=references)

predictions = ["This plugin"]
references = [
    [
        "This plugin allows you to automatically translate web pages between several languages."
    ]
]
metric.compute(predictions=predictions, references=references)

점수는 0에서 100까지 계산될 수 있으며 높을수록 좋습니다.

모델 출력에서 평가지표(metric)가 사용할 수 있는 텍스트로 변환하기 위해서 tokenizer.batch_decode() 메서드를 사용합니다. 레이블(정답)에서는 모든 -100을 제거하면 됩니다(토크나이저는 패딩 토큰에 대해 자동으로 동일한 작업을 수행함):

import numpy as np


def compute_metrics(eval_preds):
    preds, labels = eval_preds
    # 모델이 예측 로짓(logits)외에 다른 것을 리턴하는 경우.
    if isinstance(preds, tuple):
        preds = preds[0]

    decoded_preds = tokenizer.batch_decode(preds, skip_special_tokens=True)

    # -100은 건너뛴다.
    labels = np.where(labels != -100, labels, tokenizer.pad_token_id)
    decoded_labels = tokenizer.batch_decode(labels, skip_special_tokens=True)

    # 단순 후처리
    decoded_preds = [pred.strip() for pred in decoded_preds]
    decoded_labels = [[label.strip()] for label in decoded_labels]

    result = metric.compute(predictions=decoded_preds, references=decoded_labels)
    return {"bleu": result["score"]}

이제 이 작업이 완료되었으므로 모델을 미세 조정할 준비가 되었습니다!

모델을 미세 조정하기

첫 번째 단계는 Hugging Face에 로그인하는 것입니다. 이로써 결과를 Model Hub에 업로드할 수 있습니다. 주피터 노트북에는 이를 도와주는 편리한 기능이 있습니다:

from huggingface_hub import notebook_login

notebook_login()

그러면 Hugging Face 로그인 자격 증명을 입력할 수 있는 위젯이 표시됩니다.

노트북에서 작업하지 않는 경우 터미널에 다음 줄을 입력하기만 하면 됩니다:

huggingface-cli login

이 작업이 완료되면 Seq2SeqTrainingArguments를 정의할 수 있습니다. Trainer와 마찬가지로 몇 가지 필드가 더 포함된 TrainingArguments의 하위 클래스를 사용합니다:

from transformers import Seq2SeqTrainingArguments

args = Seq2SeqTrainingArguments(
    f"marian-finetuned-kde4-en-to-fr",
    evaluation_strategy="no",
    save_strategy="epoch",
    learning_rate=2e-5,
    per_device_train_batch_size=32,
    per_device_eval_batch_size=64,
    weight_decay=0.01,
    save_total_limit=3,
    num_train_epochs=3,
    predict_with_generate=True,
    fp16=True,
    push_to_hub=True,
)

일반적인 하이퍼파라미터(예: 학습률, 에포크 수, 배치 크기 및 일부 가중치 감소)를 제외하고, 다음은 이전 섹션에서 본 것과 비교하여 몇 가지 변경 사항입니다:

  • 평가에 시간이 걸리므로 정기적인 평가를 설정하지 않습니다. 우리는 학습 전과 후에 모델을 한 번만 평가할 것입니다.

  • fp16=True로 설정하여 최신 GPU에서 학습 속도를 높입니다.

  • 위에서 설명한 것처럼 predict_with_generate=True로 설정합니다.

  • push_to_hub=True를 사용하여 각 에포크(epoch)가 끝날 때 Hub에 모델을 업로드합니다.

hub_model_id 인수를 사용하여 푸시하려는 저장소의 전체 이름을 지정할 수 있습니다(특히, 조직에 푸시하려면 이 인수를 사용해야 함). 예를 들어, 모델을 huggingface-course 조직에 푸시할 때 hub_model_id="huggingface-course/marian-finetuned-kde4-en-to-fr"을 Seq2SeqTrainingArguments에 추가하면 됩니다. 기본적으로 사용되는 저장소는 네임스페이스에 존재하고 설정한 출력 디렉토리의 이름을 따서 명명되므로 이 경우에는 "spasis/marian-finetuned-kde4-en-to-fr"(우리가 이 섹션의 시작 부분에서 링크한 모델)이 됩니다.

💡 사용하는 출력 디렉토리가 이미 존재하는 경우 푸시하려는 저장소의 로컬 클론이어야 합니다. 그렇지 않은 경우 Seq2SeqTrainer를 정의할 때 오류가 발생하기 때문에 새로운 이름을 설정해야 합니다.

마지막으로 모든 것을 Seq2SeqTrainer에 전달합니다.

from transformers import Seq2SeqTrainer

trainer = Seq2SeqTrainer(
    model,
    args,
    train_dataset=tokenized_datasets["train"],
    eval_dataset=tokenized_datasets["validation"],
    data_collator=data_collator,
    tokenizer=tokenizer,
    compute_metrics=compute_metrics,
)

학습하기 전에 먼저 초기 모델이 얻는 점수를 살펴보고, 미세 조정으로 상황을 악화시키고 있지 않은지 다시 확인합니다. 이 명령은 다소 시간이 걸립니다:

trainer.evaluate(max_length=max_target_length)

BLEU 점수 39는 그리 나쁘지 않은데, 이는 우리가 선택한 모델이 이미 영어 문장을 프랑스어 문장으로 번역하는데 효과적이다는 사실을 반영합니다.

다음으로 본격적으로 학습을 진행합니다. 역시 시간이 좀 걸립니다:

trainer.train()

학습이 진행되는 동안 모델이 저장될 때마다(여기서는 모든 에포크마다) 백그라운드에서 모델이 허브에 업로드됩니다. 이런 방식으로 필요한 경우 다른 머신에서 학습을 재개할 수 있습니다.

학습이 완료되면 모델을 다시 평가해 봅시다. BLEU 점수가 개선되었기를 바랍니다!

trainer.evaluate(max_length=max_target_length)

거의 14포인트 개선되었네요. 좋습니다.

마지막으로, push_to_hub() 메서드를 사용하여 최신 버전의 모델을 업로드했는지 확인합니다. Trainer는 모든 평가 결과가 포함된 모델 카드의 초안을 작성하여 업로드합니다. 이 모델 카드에는 Model Hub가 추론 데모용 위젯을 선택하는데 도움이 되는 메타데이터가 포함되어 있습니다. 일반적으로 모델 클래스에서 올바른 위젯을 유추할 수 있으므로 아무 것도 할 필요가 없지만, 이 경우 동일한 모델 클래스를 모든 종류의 sequence-to-sequence 문제에 사용할 수 있으므로 번역 모델이라고 지정합니다:

trainer.push_to_hub(tags="tanslation", commit_message="Training complete")

위 명령은 방금 수행한 커밋의 URL을 반환합니다.max_target_length서 Model Hub의 추론 위젯을 사용하여 모델을 테스트하고 친구와 공유할 수 있습니다. 번역 작업을 위해서 모델을 성공적으로 미세 조정했습니다. 축하합니다!

학습 루프에 대해 좀 더 자세히 알고 싶다면 이제 🤗Accelerate를 사용하여 동일한 작업을 수행하는 방법을 보여드리겠습니다.

맞춤형 학습 루프 (Custom Training Loop)

이제 전체 학습 루프(full training loop)를 살펴보고 필요한 부분을 쉽게 커스터마이징할 수 있습니다. 7.2와 7.3에서 했던 것과 매우 유사할 것입니다.

학습을 위한 모든 사항 준비하기

아래 내용은 이미 몇번씩 공부했으므로 코드를 매우 빠르게 살펴보겠습니다. 먼저 데이터셋을 "torch" 형식으로 설정한 후 데이터셋에서 DataLoader를 빌드하여 PyTorch 텐서를 얻습니다:

from torch.utils.data import DataLoader

tokenized_datasets.set_format("torch")
train_dataloader = DataLoader(
    tokenized_datasets["train"],
    shuffle=True,
    collate_fn=data_collator,
    batch_size=8,
)
eval_dataloader = DataLoader(
    tokenized_datasets["validation"], 
    collate_fn=data_collator, 
    batch_size=8
)

다음으로 이전의 미세 조정을 계속해서 수행하지 않고 사전 학습된 모델에서 다시 시작하도록 모델을 다시 인스턴스화합니다:

model = AutoModelForSeq2SeqLM.from_pretrained(model_checkpoint)

이제 옵티마이저(optimizer)가 필요합니다:

from transformers import AdamW

optimizer = AdamW(model.parameters(), lr=2e-5)

지금까지 준비된 모든 객체를 Accelerator.prepare() 메서드에 보낼 수 있습니다. Colab 노트북에서 TPU를 이용하여 학습하려면 이 모든 코드를 학습 함수로 이동해야 하며 Accelerator를 인스턴스화하는 셀을 실행해서는 안됩니다.

from accelerate import Accelerator

accelerator = Accelerator()
model, optimizer, train_dataloader, eval_dataloader = accelerator.prepare(
    model, optimizer, train_dataloader, eval_dataloader
)

이제 train_dataloader를 accelerator.prepare()로 보냈으므로 길이를 사용하여 학습 단계 수를 계산할 수 있습니다. accelerator.prepare()가 DataLoader의 길이를 변경하므로 항상 dataloader를 준비한 후에 이 작업을 수행해야 합니다. 지정된 learning rate에서 0으로 변화되는 전통적인 선형 학습 스케줄러를 사용합니다:

from transformers import get_scheduler

num_train_epochs = 3
num_update_steps_per_epoch = len(train_dataloader)
num_training_steps = num_train_epochs * num_update_steps_per_epoch

lr_scheduler = get_scheduler(
    "linear",
    optimizer=optimizer,
    num_warmup_steps=0,
    num_training_steps=num_training_steps,
)

마지막으로 모델을 Hub로 푸시하려면 작업 폴더에 Repository 객체를 생성해야 합니다. 아직 로그인하지 않았다면 먼저 Hugging Face Hub에 로그인합니다. 모델에 부여하려는 모델 ID를 기반으로 저장소 이름을 결정합니다. repo_name을 원하는 대로 자유롭게 지정해도 상관없습니다. 사용자 이름만 포함하면 get_full_repo_name()이 알아서 이름을 출력합니다:

from huggingface_hub import Repository, get_full_repo_name

model_name = "marian-finetuned-kde4-en-to-fr-accelerate"
repo_name = get_full_repo_name(model_name)
repo_name

그런 다음 해당 저장소를 로컬 폴더에 복제할 수 있습니다. 이미 존재하는 경우 이 로컬 폴더는 작업 중인 리포지토리의 복제본이어야 합니다:

output_dir = "marian-finetuned-kde4-en-to-fr-accelerate"
repo = Repository(output_dir, clone_from=repo_name)

이제 repo.push_to_hub() 메서드를 호출하여 output_dir에 저장한 모든 것을 업로드할 수 있습니다. 이렇게 하면 각 에포크가 끝날 때 중간 모델을 업로드하는 데 도움이 됩니다.

학습 루프 (Training loop)

이제 전체 학습 루프를 작성할 준비가 되었습니다. 평가 부분을 단순화하기 위해 예측과 레이블을 가져와 메트릭 객체가 예상하는 문자열 목록으로 변환하는 postprocess() 함수를 정의합니다:

def postprocess(predictions, labels):
    predictions = predictions.cpu().numpy()
    labels = labels.cpu().numpy()

    decoded_preds = tokenizer.batch_decode(predictions, skip_special_tokens=True)

    # Replace -100 in the labels as we can't decode them
    labels = np.where(labels != -100, labels, tokenizer.pad_token_id)
    decoded_labels = tokenizer.batch_decode(labels, skip_special_tokens=True)

    # Some simple post-processing
    decoded_preds = [pred.strip() for pred in decoded_preds]
    decoded_labels = [[label.strip()] for label in decoded_labels]
    return decoded_preds, decoded_labels

학습 루프는 평가 부분에서 몇가지 차이점을 제외하고 섹션 2 및 3의 루프와 매우 유사하므로 이 차이점에 초점을 맞추겠습니다!

가장 먼저 주목해야 할 점은 예측을 수행하기 위해 generate() 메서드를 사용한다는 것입니다. 그러나 이것은 기본 모델의 메서드입니다. 🤗Accelerate가 prepare() 메서드에서 생성한 래핑된 모델이 아닙니다. 그래서 먼저 모델의 래핑 부분을 풀고 이 메서드를 호출합니다.

두 번째 차이점은 토큰 분류와 마찬가지로 두 개의 프로세스가 입력과 레이블을 다른 모양으로 패딩했을 수 있으므로 gather() 메서드를 호출하기 전에 Accelerator.pad_across_processes()를 사용하여 예측 결과와 레이블을 동일한 모양으로 만듭니다. 이렇게 하지 않으면 평가에 오류가 발생하거나 프로세스가 영원히 중단됩니다.

from tqdm.auto import tqdm
import torch
import numpy as np

progress_bar = tqdm(range(num_training_steps))

for epoch in range(num_train_epochs):
    # 학습
    model.train()
    for batch in train_dataloader:
        outputs = model(**batch)
        loss = outputs.loss
        accelerator.backward(loss)

        optimizer.step()
        lr_scheduler.step()
        optimizer.zero_grad()
        progress_bar.update(1)

    # 평가
    model.eval()
    for batch in tqdm(eval_dataloader):
        with torch.no_grad():
            generated_tokens = accelerator.unwrap_model(model).generate(
                batch["input_ids"],
                attention_mask=batch["attention_mask"],
                max_length=128,
            )
        labels = batch["labels"]

        # 예측과 레이블을 모으기 전에 함께 패딩 수행
        generated_tokens = accelerator.pad_across_processes(
            generated_tokens, dim=1, pad_index=tokenizer.pad_token_id
        )
        labels = accelerator.pad_across_processes(labels, dim=1, pad_index=-100)

        predictions_gathered = accelerator.gather(generated_tokens)
        labels_gathered = accelerator.gather(labels)

        decoded_preds, decoded_labels = postprocess(predictions_gathered, labels_gathered)
        metric.add_batch(predictions=decoded_preds, references=decoded_labels)

    results = metric.compute()
    print(f"epoch {epoch}, BLEU score: {results['score']:.2f}")

    # 저장 및 업로드
    accelerator.wait_for_everyone()
    unwrapped_model = accelerator.unwrap_model(model)
    unwrapped_model.save_pretrained(output_dir, save_function=accelerator.save)
    if accelerator.is_main_process:
        tokenizer.save_pretrained(output_dir)
        repo.push_to_hub(
            commit_message=f"Training in progress epoch {epoch}", blocking=False
        )

이 작업이 완료되면 Seq2SeqTrainer로 학습된 모델과 매우 유사한 결과를 갖는 모델을 갖게 됩니다. huggingface-course/marian-finetuned-kde4-en-to-fr-accelerate에서 이 코드를 사용하여 훈련한 것을 확인할 수 있습니다. 그리고 위 코드를 편집하여 학습 루프를 수정하고 테스트해 볼 수 있습니다.

미세 조정된 모델 사용하기

앞에서 추론 위젯을 사용하여 Model Hub에서 미세 조정한 모델을 사용하는 방법을 이미 살펴봤습니다. 파이프라인을 활용하여 로컬로 사용하려면 적절한 모델 식별자를 지정하기만 하면 됩니다:

from transformers import pipeline

model_checkpoint = "huggingface-course/marian-finetuned-kde4-en-to-fr"
translator = pipeline("translation", model=model_checkpoint)
translator("Default to expanded threads")

예상대로, 사전 학습된 모델은 해당 지식을 우리가 미세 조정한 말뭉치에 적용했으며 영어 단어 "threads"를 그대로 두는 대신 이제 프랑스어 공식 버전으로 번역합니다. "plugin"에 대해서도 동일합니다:

translator(
    "Unable to import %1 using the OFX importer plugin. This file is not the correct format."
